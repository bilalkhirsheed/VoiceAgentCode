--- 1) Why "technical issue" / transfer fails when testing static number ---

If you use a static number (e.g. +923244983583) and the flow always goes to "Transfer failed" → "we have a technical issue", common causes:

- Retell/carrier cannot reach that number (e.g. region not supported, number not verified, or blocked).
- Number format: use E.164 with country code (e.g. +923244983583). Enable "Format to E.164" in the node.
- Test Agent (Simulation): transfer often does not actually connect in simulation; the engine may treat it as "failed" and follow the "Transfer failed" transition. So you may see "technical issue" even when the config is correct.
- Real phone call: to test static transfer for real, place a real call to your Retell number (from another phone), go through the flow until the transfer runs, and ensure the destination number is a real, reachable phone (same country as your Retell number is usually more reliable). If it still fails, check Retell dashboard logs for the transfer failure reason.

To test static transfer without hitting "technical issue" in Simulation: you can temporarily change the "Transfer failed" transition so it does not go to the "technical issue" conversation (e.g. disconnect or retry), or accept that Simulation may not complete the transfer and test with a real call instead.

  A call appears here when the call is categorized as "callback". The backend sets this when Retell sends call_analyzed with EITHER:
  - custom_analysis_data.Call_Back_Capture set (e.g. true or a value), OR
  - collected_dynamic_variables.Call_Back_Capture set, OR
  - Reason_for_call (or service_request) containing the word "callback".

User Hangups (CRM → User Hangups):
  A call appears here when the caller (user) hung up. The backend sets this when Retell sends call_analyzed with:
  - disconnection_reason containing "user" (e.g. "user_hangup", "user_hangup_timeout").
  So: any call that ends because the user hung up (not the agent or system) will have is_user_hangup = true and show under User Hangups.

Note: A single call can be both a callback and a user hangup (e.g. user requested callback then hung up). It will then appear in BOTH Callback Requests and User Hangups. The dashboard uses call_analysis.category for "callback" and call_analysis.is_user_hangup for "User Hangups".
